toy's Standardised Versioning

Written: 2024-06-17 · Last Updated: Never

After seven years or so of releasing software, I felt it was time to address a common problem in my personal programming style: the lack of a consistent versioning standard. Probably the most common versioning system for a long time now is Semantic Versioning, or SemVer.

SemVer, overall, is a pretty decent system; however, it has one main issue: its design target. SemVer is for stuff like libraries and services—things that provide APIs. For that purpose, it works pretty well and helps devs avoid Dependency Hell and its related issues by establishing a kind of paradigm for releasing software that enforces a standard of mutually understood versioning between developers and the other developers that use their API. But my main issue here is that I've never used npm, I've never spun up a Node instance, and I've never had to deal with dependency chains. I'm not a Web dev, I'm a desktop programmer at heart. My applications and coding practices are a different world than that which SemVer was designed for.
So, for ages I've basically just ad-libbed my versioning from project to project, making it up as I go. This is not entirely bad for the fact that nothing has ever had to actually depend on my code, but it does suck that every time I switch project I have to go check the GitHub releases tab to see how I was formatting things.

I have a couple goals for my standard:

  1. Be stricter and more concise than SemVer: SemVer has what it calls "pre-release version" and "metadata" strings than can even be used simultaneously, such as 1.0.0-alpha+001.1 I think is not strict enough for my use case, and there are better, more concise ways to convey meaning in my version strings.
  2. Establish a strict and upholdable standard for myself: I want something with a clear and concise reference that not only I, but anyone else can also apply to their general software production practice.

…And so, I present you with:

toy's Standardised Versioning

(Working Title)

Format: <ring><major>.<minor>[.<patch>][_hotfix|-rc]

  • <ring>: A "ring" is a bit of Microsoft terminology that I'm going to repurpose here.2 By "ring," I'm referring to Alpha/Beta/Release channels throughout your software development process, which I will define as follows:
    • Alpha is when you're right at the start of the project and you're primarily focused on completing full functionality to your application. The string for the alpha phase is a.
    • Beta is the bug-fixing process and the point in time when edge-case testing etc occurs. This might also be the time when you open an early access preview to your software. The string for beta is b.
    • Release is the production copy of your software: fully functional and free of bugs. Release versions use the string v (for "version").
  • <major>: The major version number always starts from 1, even in alpha. A major version of 0 is not allowed. This number is incremented when your program implements a breaking incompatibility (i.e: dropping support for an old config file format; but not when config specifications are changed but the program retains functionality to update configs to the new format) or when a complete refactor/redesign of the application occurs that changes the underlying method in which your software works. In short: this number is rarely updated—it might even stay at 1 for over a decade! It doesn't increment like Firefox or Chrome major version 7 billion releases every month.
  • <minor>: A minor update would usually be something like a feature update that adds or removes a chunk of functionality to/from your program. This number starts at 0 whenever the major release is incremented, and can increment indefinitely. (In this case, it can increment like a Firefox major version does, if your project is that big, then I'd expect a version number like v1.159 to come around after 10 or 15 years easy)
  • [patch]: Patches are purely bugfix updates and shouldn't (but not strictly) change any functionality whatsoever. Each patch update starts at 0 when the minor version is incremented, and can go indefinitely. Patch updates should usually be shipped as a bundle of bugfixes per patch, rather than batching the bug-fixing process into multiple patches.
  • [_hotfix|-rc]: The final part of the version is two mutually exclusive options:
    • [hotfix]: A hotfix is a hastily created update with a small number (usually just one) of bug fixes. The kind of thing you'd ship on short notice if your latest version has a critical bug in it that can't wait until the next patch. Generally, I keep hotfix numbers two digits long at least, so if I were to patch v1.2, then the update would be v1.2_01—then I could release _02, _03, etc indefinitely (including adding more digits, but over 99 hotfixes to one update is some pretty embarrasing code writing). Hotfixes should not be applied to alpha and beta ring versions, but this is not a strict rule.
    • [rc]: "RC" stands for release candidate. It is mutually exclusive to hotfixes because a release candidate is a non-production piece of software that the user, in downloading your software, is knowingly accepting its instability. Release candidates are also how you distribute early access releases once you are past the v1.0 version number. Rather than implement a more complex system (as one of my goals is to be strict and concise), a release candidate system is all I will add for already released software's early access channels.
      Say you're at v1.2.3, and you want to release a preview for update v1.3; the version would be v1.3-rc1. You can then release an rc2 and rc3, as many as you'd like, until the real v1.3 is released. Also, as your rcs are pre-releases, you can continue deploy an update to the regular channel like update v1.2.4.

Example of toy's Standardised Versioning: Through a Software Lifecycle

  1. a1.0: The project is created, manifests/makefiles/build tools/documentation/code structure is set up.
  2. a1.1: Some more code is added to do more things.
  3. b1.0: The program can do everything it is designed to do, but very likely not reliably nor efficiently.
  4. b1.0.1: b1.0 had a couple bugs in it that have now been fixed.
  5. (Skipping ahead a bit) b1.5.3: Bugfixing and optimisations have changed a lot of the structure of the code and its algorithms are likely very different, if not indistinguishable, from late alpha/b1.0.
  6. v1.0: The fully-functional, bug-free software is released!
  7. v1.1: Some new function is added.
  8. v1.1_01: The new function had an unhandled exception which was not caught and unfortunately caused user's PCs to combust—which has now been fixed.
  9. (Skipping a bit more) v1.3.7: The software now does a few new things and is overall more stable due to bugfixes. The names of some options in the config files have changed since v1.0, but functions exist to update the config files of users with outdated configs. One function of the software has also been removed.
  10. v2.0-rc1: Based on the v1.3.7 codebase, a major refactor occurs and the pre-v1.3.7 config specification is discarded, meaning no v1.0–v1.36-created config files will work in the software anymore. Additionally, some functions have been added and some have been removed.
  11. v2.0-rc2: This update changes one line of code because the last update wouldn't launch on certain devices
  12. (Skipping) v2.0-rc10: Numerous changes to functions, bugfixes, and compatibility have been made
  13. v2.0: A full, stable release (based on the v2.0-rc10 codebase) releases to succeed the last general release, v1.3.7.
